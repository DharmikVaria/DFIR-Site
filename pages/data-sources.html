Artifact Layer

Digital Evidence Sources and Artifacts

Artifacts are traces created by operating systems, applications, users, malware, services, and networks. Strong investigations correlate artifacts across multiple sources to avoid weak conclusions.

Why artifact sources matter

A single event can leave traces in many places. A downloaded executable may appear in browser history, download cache, Zone.Identifier alternate data streams, Prefetch, Event Logs, registry user assist keys, antivirus logs, filesystem metadata, memory, and network records. Correlating these traces improves confidence and reduces false assumptions.

Artifact analysis is about context. Investigators ask what created the artifact, how reliable the timestamp is, whether the artifact can be modified, whether it identifies a user or only a system, and what additional evidence can confirm it.

Windows artifacts

  • MFT: NTFS Master File Table records filenames, metadata, timestamps, and deleted file traces.
  • Registry: Stores system, user, application, USB, network, and persistence information.
  • Prefetch: Indicates program execution, run count, and related file references.
  • Event Logs: Capture logons, process events, service activity, PowerShell, security events, and system changes.
  • LNK files: Shortcut artifacts that can show opened files, paths, volumes, and timestamps.
  • Jump Lists: Recent and frequent application activity, useful for user behavior reconstruction.
  • Shellbags: Folder viewing history, including removable or deleted directory references.
  • Recycle Bin: Deleted file names, original paths, deletion timestamps, and user SID references.
  • SRUM: System Resource Usage Monitor data showing application resource and network usage.

Linux artifacts

  • /var/log/auth.log or secure: SSH logins, sudo use, authentication failures, and privilege activity.
  • syslog and messages: System service events, daemon output, kernel messages, and operational traces.
  • bash history: Commands executed by users, though it may be disabled, edited, or incomplete.
  • cron: Scheduled tasks used for administration or persistence.
  • systemd journals: Structured logs for services, boots, units, and system activity.
  • SSH artifacts: authorized_keys, known_hosts, config files, and login traces.
  • Package logs: apt, yum, dnf, dpkg, and rpm logs showing installed or removed software.

Network artifacts

Network evidence helps identify communication patterns, command-and-control activity, data transfer, scanning, brute force attempts, DNS tunneling, and suspicious endpoints. It is especially useful when host logs are missing or tampered with.

  • PCAP: Packet-level evidence that can reveal protocols, payloads, sessions, and files.
  • DNS logs: Domain lookups, suspicious dynamic domains, tunneling, and malware beacons.
  • Proxy logs: Web requests, user agents, downloaded files, URLs, and authenticated users.
  • Firewall logs: Allowed and blocked connections, ports, IPs, and traffic direction.
  • Zeek logs: Structured network metadata for connections, DNS, HTTP, SSL, files, and notices.

Cloud artifacts

Cloud forensics focuses on audit trails, identity activity, storage access, API calls, configuration changes, and data movement. Investigators must act quickly because log retention may be limited by plan, configuration, or tenant policy.

  • AWS CloudTrail: API calls, IAM activity, console logins, role use, and resource changes.
  • Azure logs: Entra ID sign-ins, audit logs, activity logs, conditional access, and resource operations.
  • GCP audit logs: Admin activity, data access events, and service operations.
  • M365 audit logs: Exchange, SharePoint, OneDrive, Teams, mailbox access, file sharing, and user activity.

Memory artifacts

Memory analysis reveals runtime state that may not exist on disk. Investigators can identify suspicious processes, injected code, loaded DLLs, open handles, command lines, credentials, network connections, unpacked malware, and encryption material.

Tools such as Volatility 3 help parse memory images into process lists, network structures, registry hives, command history, modules, and suspicious memory regions. Findings should be correlated with disk and log evidence.

Mobile artifacts

Mobile devices contain communication, location, application, media, and account evidence. Acquisition depth depends on device lock state, encryption, OS version, legal authority, and available tools.

  • SMS and calls: Message content, call duration, numbers, timestamps, and contact links.
  • App databases: SQLite databases for chats, social apps, browsers, and local app state.
  • Location data: GPS, Wi-Fi, cell tower, map history, and application location caches.
  • EXIF metadata: Camera model, timestamps, GPS coordinates, and editing traces in images.
  • Tokens: Authentication artifacts that may show account access or session state.

Real investigation usage

In an insider data theft case, investigators may combine Windows Shellbags, LNK files, USB registry keys, cloud sync logs, browser uploads, archive creation timestamps, and proxy logs. In ransomware response, they may correlate process execution, scheduled tasks, event logs, file rename patterns, ransom notes, memory evidence, and firewall connections.

The strongest findings come from convergence: multiple independent artifacts pointing to the same event.

Beginner artifact mistakes

  • Assuming one artifact proves the entire event.
  • Ignoring timestamp source, time zone, and last modified behavior.
  • Not checking whether logs were cleared, rotated, or overwritten.
  • Confusing account activity with human attribution.
  • Failing to validate automated parser output against raw artifacts.
  • Collecting artifacts without preserving source path and context.
