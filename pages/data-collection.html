Acquisition Layer

Data Collection: Live, Dead, and Triage Forensics

Collection strategy determines what evidence survives and how reliable later analysis will be. Investigators must balance evidence preservation, business urgency, volatility, encryption, malware behavior, and legal authority.

What data collection means

Data collection is the process of acquiring digital evidence from systems, storage devices, applications, networks, cloud services, and memory. The goal is to obtain data in a manner that preserves integrity and supports later examination.

Collection is not a one-size-fits-all step. A ransomware investigation, insider theft case, malware infection, cloud account compromise, and mobile device examination may each require different acquisition priorities.

Live forensics

Live forensics is performed while a system is powered on and running. It is used when volatile evidence may disappear if the system is shut down. Examples include RAM contents, running processes, active network connections, logged-in users, mounted encrypted volumes, temporary files, malware running only in memory, and unsaved command output.

Live collection is useful but invasive. Every command executed can alter memory, process state, logs, shell history, filesystem metadata, and network activity. Therefore, live response must be purposeful, minimal, and carefully documented.

Dead or static forensics

Dead forensics is performed on a powered-off system or detached storage device. The examiner images the disk using a forensic workstation, preferably with a write blocker. Analysis is then performed on the image, not the original media.

Static acquisition is ideal when volatile evidence is not critical or when preserving the storage state is the top priority. It is commonly used for deleted file recovery, timeline analysis, file system examination, registry analysis, browser history, and user activity reconstruction.

Order of volatility

The order of volatility ranks evidence by how quickly it can change or disappear. Highly volatile data should be collected first when it is relevant and legally authorized.

  1. CPU registers and cache data.
  2. RAM, running processes, open sockets, logged-in users.
  3. Temporary filesystem data and swap/pagefile content.
  4. Local disk data and application artifacts.
  5. Remote logs, cloud logs, backups, and archived data.

RAM capture

RAM can reveal processes, injected code, network connections, encryption keys, command history, credentials, malware unpacked in memory, and artifacts that never touched disk. Tools such as WinPmem, Magnet RAM Capture, DumpIt, Belkasoft RAM Capturer, and Linux memory acquisition methods may be used depending on system type and policy.

Memory capture must be done before shutdown. The investigator should record tool name, version, command line, start time, end time, output hash, and storage destination. Capturing memory to the suspect disk should be avoided when possible because it can overwrite evidence.

Disk imaging

Disk imaging creates a sector-by-sector or logical copy of storage media. A physical image captures the full device, including unallocated space and slack space. A logical image captures selected files and folders, usually when full imaging is impractical.

Full physical images are valuable because they preserve deleted files, filesystem metadata, partition structures, volume shadow copies, and hidden areas. Logical collection is faster but may miss deleted data and low-level artifacts.

Forensic image formats

RAW / ddSimple bitstream image. Widely supported, but lacks rich metadata and compression.
E01Expert Witness format. Supports compression, metadata, segmentation, and integrity checks.
AFFAdvanced Forensic Format. Open format designed to store image data and metadata.

The best format depends on tool compatibility, case requirements, storage capacity, verification needs, and organizational policy.

Hash verification

Hash verification confirms that the acquired image matches the source data as captured. Investigators should calculate and record hash values during acquisition and after transfer. If segmented images are used, the tool should verify the entire image set.

A mismatch does not automatically prove tampering, but it must be investigated. Causes may include acquisition error, device instability, write access, incomplete copy, bad sectors, or wrong source media.

Triage collection

Triage collection gathers high-value evidence quickly when full imaging is too slow or disruptive. It may collect event logs, registry hives, browser data, prefetch files, user folders, running process data, network connections, persistence locations, and recent file activity.

Triage is useful during enterprise incidents where many systems must be assessed rapidly. It should not be confused with complete forensic acquisition. It is a prioritization method, not a replacement for deeper analysis when the case requires it.

When not to shut down a system

  • Full disk encryption is active and keys may only exist in memory.
  • Malware is memory-resident and may disappear after power loss.
  • Active network connections may identify command-and-control infrastructure.
  • Logged-in cloud sessions or mounted remote drives may be relevant.
  • Running processes, open files, or volatile logs are essential to the case question.

Risks of altering evidence

Evidence alteration can occur through booting, logging in, opening files, copying data incorrectly, installing tools, writing output to the suspect disk, or allowing automated updates to run. Even well-intentioned actions can change timestamps and logs.

When change is unavoidable, the answer is not to hide it but to document it. Record the reason, action taken, time, tool, expected impact, and any observed system changes.

Beginner collection mistakes

  • Turning off a system before considering encryption or volatile evidence.
  • Capturing memory after running many unnecessary commands.
  • Storing acquisition output on the suspect drive.
  • Forgetting to hash evidence and acquired images.
  • Using logical copies when deleted data or filesystem metadata matters.
  • Failing to document bad sectors, acquisition errors, or tool warnings.
